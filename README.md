## 🚀 開発の始め方（環境構築）

作業を始める前に、以下の手順でローカル環境を立ち上げてください。

**1. (初回のみ)リポジトリをクローンしてフォルダに移動する**

```bash
git clone [リポジトリのURL] [(任意)作業パス]
cd [作業パス]
```

**2 環境変数の準備**

```bash
cp .env.example .env
```

※.envに記載するパスワードは個別で共有すること

**2 Dockerでサーバを立ち上げる**

```bash
docker-compose up -d
```

**3 起動確認**

⭐️バックエンド

ブラウザで `http://localhost:8000` にアクセスし、

`Hello Hono!`と表示されれば準備完了です！

⭐️フロントエンド

ブラウザで `http://localhost:3000` にアクセスし、

`Next.jsのページ`が表示されれば準備完了です！

⭐️DB

Prisma StudioでDBのテーブルを確認したい場合は、以下コマンドを実行する。

```bash
docker-compose exec backend npx prisma studio --browser none --hostname 0.0.0.0
```

起動後、ブラウザで `http://localhost:5555` を開いてください。

**4 コンテナ停止**

`docker-compose down` で停止します。

# 🐈‍⬛ Git Rules🐈‍⬛

## ⭐️作業時のルール

作業を始める時は、最新の `main` からブランチを作成して作業をお願いします。

ブランチの命名規則は以下を参照。

また、自分が作業している間に、他の人のプルリクが `main` にマージされた場合、

自分のブランチに最新の `main` を取り込んでからプルリクエストを出してください。

※この時点でコンフリクトが起きている場合は、確認すること。

🔻メインブランチの最新状態を取り込むコマンド(作業ブランチから実行)

```
 git pull origin main
```

## 📕 ブランチ名のルール

ブランチ名は 「種類/やること」 で統一します。

feature/：新しい機能を作るとき

fix/：バグやエラーを直すとき

docs/：ドキュメント（READMEなど）だけを更新するとき

chore/：細かい修正

## ✍🏻 コミットメッセージのルール

コミットメッセージは 「種類: 何をしたか」 で統一します。

レビュー者も確認しやすいため、こまめにコミットするようにしましょう！

`feat: 新しい機能を追加`

例：feat: LINEからのメッセージを受け取る処理を追加

`fix: バグを修正した`

例：fix: 空文字を送るとサーバーが落ちるバグを修正

`docs: ドキュメントを更新した`

例：docs: READMEに環境構築の手順を追記

`chore: 設定ファイルなどの裏側の作業をした`

例：chore: 必要なPythonライブラリ（requirements.txt）を追加

## 🌼 プルリクエストのルール

### 👷 _作業者_

作業の区切りが良いタイミングでプルリクエストを作成する。

プルリクエスト時のタイトルはブランチ名ルール同様、「種類: 取り込んだこと」 で統一します。

プルリクエスト時のコメントはテンプレートに沿って記載すること。

併せて、タスク管理(notion)のタスクビューのPRレビューグループに新規ページを追加して、以下を設定する。

`ページ名「【レビュー】プルリクエストタイトル名」`

`カテゴリ「PRレビュー」`

`ステータス「未着手」`

`GitHubPR「プルリクエストURL」`

上記完了後、チームチャットにレビュー依頼を送ること。

### 🧑‍🏫 _レビュー者_

レビュー可能な担当者は、タスク管理(notion)のタスクビューのPRレビューグループを確認する。

対象のプルリクエストを確認し、担当者を自分、ステータスを「進行中」に変更する。

差分内容を確認し、問題なければmainにマージする。

上記でプルリク対応は完了のため、notionページのステータスは「完了」に変更すること。
